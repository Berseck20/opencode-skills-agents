## What I do  

- Parse `audit-report-[business-slug].md` and `perf-a11y-report-[business-slug].md`  
- Build a prioritized fix queue ordered by severity  
- Apply every fix directly to the corresponding project file  
- Verify each fix doesn't introduce regressions  
- Produce a final resolution log confirming every issue is resolved or documenting why it wasn't  
  
## When to use me  
  
Use this skill as the third and final step in the `error-finder` workflow, after both `code-audit` and `perf-a11y` have produced their reports.  
  
Requires:  
- `audit-report-[business-slug].md`  
- `perf-a11y-report-[business-slug].md`  
- Complete Astro project (post-rebuild)  
  
## Fix priority queue  
  
Process issues in this exact order:  
  

    🔴 CRITICAL — code-audit (build won't work)
    🔴 CRITICAL — perf-a11y (section inaccessible / page unusable)
    🟠 HIGH — code-audit (visible breakage)
    🟠 HIGH — perf-a11y (a11y violation / performance failure)
    🟡 MEDIUM — code-audit (visual inconsistency)
    🟡 MEDIUM — perf-a11y (optimization gap)
    🔵 LOW — all sources

Within the same severity, order by:  
1. Data binding fixes first (prevent blank sections)  
2. HTML structure fixes (prevent rendering issues)  
3. CSS fixes (prevent visual breakage)  
4. Accessibility fixes (prevent legal risk)  
5. Performance fixes (prevent slow experience)  
6. Visual consistency fixes (prevent quality perception drop)  
  
## Fix application protocol  
  
### Step 1 — Parse reports  
Read both audit reports and extract every issue into a structured queue:  
  
```markdown  
[QUEUE-001] 🔴 CRITICAL | code-audit | ISSUE-003  
  File: src/components/sections/Hero.astro  
  Line: 24  
  Issue: Broken data import — `businessData` is undefined  
  Fix: Change import path from `../../data/business` to `../../data/business.json`  
  
[QUEUE-002] 🔴 CRITICAL | perf-a11y | A11Y-001  
  File: src/layouts/Layout.astro  
  Line: 1  
  Issue: Missing lang attribute on <html>  
  Fix: Add lang="en" to <html> tag  

Count total issues and report before starting:

Total issues queued: [X]  
  🔴 CRITICAL: [X]  
  🟠 HIGH: [X]  
  🟡 MEDIUM: [X]  
  🔵 LOW: [X]  
Estimated fixes: [X] automated / [X] require manual review```

Step 2 — Pre-fix snapshot

Before any modifications, document the current state of every file that will be touched:

Files to modify: [count]  
- src/components/sections/Hero.astro (3 fixes)  
- src/layouts/Layout.astro (2 fixes)  
- src/styles/global.css (5 fixes)  
- src/data/business.json (1 fix)  

Step 3 — Apply fixes sequentially
For each issue in the queue:

    Read the target file
    Locate the exact line or block referenced in the report
    Verify the issue still exists (another fix may have resolved it)
    Apply the fix using surgical edits — change only what's needed
    Verify the fix doesn't break adjacent code
    Log the result

Step 4 — Cross-fix verification

After all fixes are applied, run these cross-checks:

| Check             | Purpose                                                 |
|-------------------|---------------------------------------------------------|
| Import chain      | All imports still resolve after path changes            |
| Data bindings     | All component → data connections still work             |
| Class references  | No Tailwind classes were orphaned or duplicated         |
| Design tokens     | All var(--*) still reference defined properties         |
| Heading hierarchy | Fix to one heading didn't break the h1→h2→h3 chain      |
| Contrast pairs    | Color fixes maintained or improved contrast ratios      |
| Loading order     | Performance fixes didn't change critical resource order |

Fix patterns

Data binding fixes

<!-- BEFORE: broken import -->  
---  
import { services } from '../../data/services'  
---  
  
<!-- AFTER: correct extension and path -->  
---  
import { services } from '../../data/services.json'  
---  

<!-- BEFORE: no null guard -->  
{business.phone}  
  
<!-- AFTER: null-safe rendering -->  
{business.phone && <a href={`tel:${business.phone}`}>{business.phone}</a>}  

HTML structure fixes

<!-- BEFORE: skipped heading level -->  
<h1>Business Name</h1>  
<h3>Our Services</h3>  
  
<!-- AFTER: correct hierarchy -->  
<h1>Business Name</h1>  
<h2>Our Services</h2>  

<!-- BEFORE: div as button -->  
<div class="cta" onclick="...">Book Now</div>  
  
<!-- AFTER: semantic button -->  
<button class="cta" type="button">Book Now</button>  

CSS fixes

/* BEFORE: hardcoded color */  
.card { border-color: #3b82f6; }  
  
/* AFTER: design token */  
.card { border-color: var(--color-primary); }  

/* BEFORE: missing fallback */  
.heading { color: var(--color-heading); }  
  
/* AFTER: with fallback */  
.heading { color: var(--color-heading, #1e293b); }  

Accessibility fixes

<!-- BEFORE: image without alt -->  
<img src="/hero.webp" />  
  
<!-- AFTER: complete image tag -->  
<img  
  src="/hero.webp"  
  alt="Interior of Smith's Dental Office showing modern treatment room"  
  width="1440"  
  height="600"  
  loading="eager"  
  fetchpriority="high"  
/>  

<!-- BEFORE: no skip link -->  
<body>  
  <header>...</header>  
  
<!-- AFTER: skip link added -->  
<body>  
  <a href="#main-content" class="sr-only focus:not-sr-only focus:absolute focus:top-4 focus:left-4 focus:z-50 focus:bg-white focus:px-4 focus:py-2 focus:rounded focus:shadow-lg">  
    Skip to main content  
  </a>  
  <header>...</header>  

/* BEFORE: no reduced motion */  
  
/* AFTER: respect user preference */  
@media (prefers-reduced-motion: reduce) {  
  *,  
  *::before,  
  *::after {  
    animation-duration: 0.01ms !important;  
    animation-iteration-count: 1 !important;  
    transition-duration: 0.01ms !important;  
    scroll-behavior: auto !important;  
  }  
}  

Performance fixes

<!-- BEFORE: render-blocking font -->  
<link href="https://fonts.googleapis.com/css2?family=Inter" rel="stylesheet" />  
  
<!-- AFTER: preloaded non-blocking font -->  
<link rel="preconnect" href="https://fonts.googleapis.com" />  
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin />  
<link  
  href="https://fonts.googleapis.com/css2?family=Inter:wght@400;500;600;700&display=swap"  
  rel="stylesheet"  
  media="print"  
  onload="this.media='all'"  
/>  

<!-- BEFORE: all images eager -->  
<img src="/service-1.webp" />  
  
<!-- AFTER: lazy below fold with dimensions -->  
<img  
  src="/service-1.webp"  
  width="400"  
  height="300"  
  loading="lazy"  
  decoding="async"  
  alt="Professional kitchen deep cleaning service"  
/>  

Contrast fixes

/* BEFORE: contrast ratio 3.2:1 (fails AA) */  
.subtitle { color: #94a3b8; } /* on white background */  
  
/* AFTER: contrast ratio 4.6:1 (passes AA) */  
.subtitle { color: #64748b; } /* on white background */  

When fixing contrast, follow these rules:

    Darken light text (preferred) rather than lightening dark backgrounds
    Stay within the design token palette — pick the nearest passing color
    If no palette color passes, create a new shade between two existing tokens
    Document the old ratio, new ratio, and the calculation

Conflict resolution

When fixes conflict with each other:

| Conflict                                              | Resolution                                                     |
|-------------------------------------------------------|----------------------------------------------------------------|
| A11y fix changes color that performance fix optimized | A11y wins — contrast is a legal requirement                    |
| HTML fix restructures element that CSS fix targeted   | Apply HTML fix first, adjust CSS fix to match new structure    |
| Two fixes target the same line                        | Merge both changes into one edit                               |
| Fix breaks a design token reference                   | Add the missing token to the design system, then apply the fix |
| Fix changes heading level that breaks hierarchy       | Adjust all subsequent headings to maintain valid hierarchy     |

Unfixable issues
Some issues cannot be automatically fixed. Document them clearly:

### [QUEUE-XXX] ⚪ CANNOT AUTO-FIX  
- **Original issue:** [description]  
- **Why it can't be auto-fixed:** [reason]  
- **Manual action required:** [what a human needs to do]  
- **Priority:** [how urgently this needs human attention]  

Valid reasons for not auto-fixing:
    Requires actual photography (image slots with placeholders)
    Requires business owner input (phone number, address, hours)
    Requires third-party account setup (Google Maps API, analytics)
    Requires DNS/hosting configuration
    Requires content decision that only the client can make

## Rules  
  
1. **Apply, don't recommend.** This skill edits files. Every fix that can be applied is applied. The output is a working project, not a to-do list.  
2. **Zero regressions.** Every fix is verified against adjacent code. If applying fix B breaks what fix A resolved, fix B must be adjusted — not applied blindly.  
3. **Severity order is absolute.** Critical issues are fixed first, always. A LOW fix never runs before all CRITICALs are resolved.  
4. **Surgical precision.** Change only the lines needed. Do not rewrite files, restructure components, or "clean up while we're here." Fix exactly what the reports flag.  
5. **Accessibility is non-negotiable.** When accessibility and aesthetics conflict, accessibility wins. When accessibility and performance conflict, accessibility wins. ADA compliance is a legal requirement for US businesses.  
6. **Transparency on failures.** If something can't be auto-fixed, say so clearly with the exact manual action needed. Never silently skip an issue.  
7. **The project must build.** After all fixes are applied, the project must compile without errors. If it doesn't, the last fix that broke it gets reverted and logged as manual.
